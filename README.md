Digital Twin Network — Phase 1: What We Built and How to Run It
We implemented a Digital Twin Network (DTN) based on the ITU-T Y.3090 architecture, as described in the IEEE Communications Magazine paper "6G Digital Twin Networks: From Theory to Practice." The system has four layers running in parallel: a physical network simulated by Mininet, a data collection pipeline, a digital twin synchronization engine, and a live web dashboard.
What's running: Mininet emulates a 3-switch, 6-host topology defined in config/topology_config.yaml. Switch s1 connects h1 and h2, s2 connects h3 and h4, s3 connects h5 and h6, with all three switches interconnected. The topology builder (physical_network/topology_builder.py) reads the YAML, constructs the network, registers it in SQLite, installs OpenFlow flood rules on all switches for full connectivity, and pre-populates ARP tables on all hosts. The data collector (data_layer/collector.py) uses nsenter to enter each Mininet host's Linux network namespace and runs real pings between all host pairs, storing latency, packet loss, and jitter in dtn_network.db every second. The sync engine (twin_core/sync_engine.py) reads the database every 5 seconds, updates the digital twin state, and flags anomalies. The Flask dashboard (dashboard/app.py) at http://localhost:5000 shows real-time Plotly charts of latency distribution, packet loss by link, and full topology info.
To run from scratch every session — always in this order:
Terminal 1: sudo mn -c && sudo python3 physical_network/topology_builder.py --config config/topology_config.yaml — wait for mininet> prompt, then type pingall to verify 0% drop.
Terminal 2: sudo python3 data_layer/collector.py --interval 1
Terminal 3: python3 twin_core/sync_engine.py
Terminal 4: python3 dashboard/app.py then open http://localhost:5000
Key fixes made during implementation: WAL mode enabled on SQLite for concurrent access; topology builder closes its DB connection before entering Mininet CLI so the collector can write freely; collector uses nsenter to ping from within Mininet host namespaces rather than the host machine; OVSController with flood flow rules (ovs-ofctl add-flow sX action=flood) replaces the default Controller which failed to install inter-switch routes; static ARP entries are pre-populated on all hosts eliminating ARP broadcast failures across switches.
Known behavior: The avg latency stat shows ~1000ms due to duplicate packets from flood mode — this is cosmetic. The min latency (~20ms same-switch, ~40ms cross-switch) reflects the true topology delay from the YAML config.
